\chapter{Análisis del problema, requisitos, planificación y presupuesto}
\label{chap:metodologia-plan}

\lettrine{L}{a} ejecución del TFG se organiza como un desarrollo incremental de producto, con validación continua sobre datos reales y simulados.

\section{Descripción general del sistema a desarrollar}

El sistema se plantea como un conjunto ciberfísico en el que intervienen adquisición de datos, comunicación, procesado de señal y visualización.
La arquitectura final debe permitir tanto el funcionamiento autónomo del teléfono móvil como la integración de un nodo embebido basado en MCU+ESP32+IMU.

\section{Requisitos funcionales}

Los requisitos funcionales principales del sistema son:

\begin{itemize}
  \item capturar datos inerciales en modo móvil;
  \item almacenar sesiones con metadatos de adquisición;
  \item procesar sesiones mediante un pipeline espectral homogéneo;
  \item visualizar serie temporal, espectro y métricas derivadas;
  \item conectarse al ESP32 y gestionar sesiones procedentes del modo embebido;
  \item comparar resultados entre plataformas con criterios comunes.
\end{itemize}

\section{Requisitos no funcionales}

Además de las funciones visibles para el usuario, el diseño debe satisfacer criterios de:

\begin{itemize}
  \item fiabilidad en sesiones largas;
  \item modularidad entre adquisición, procesado y visualización;
  \item portabilidad del flujo de trabajo entre plataformas;
  \item facilidad de uso en campo o en laboratorio;
  \item bajo coste y viabilidad de despliegue.
\end{itemize}

\section{Restricciones técnicas}

Las restricciones más relevantes del proyecto son:

\begin{itemize}
  \item limitaciones de frecuencia efectiva y jitter en Android;
  \item disponibilidad y comportamiento real de sensores inerciales integrados;
  \item limitaciones de memoria, conectividad y temporización en ESP32;
  \item dificultad práctica de integración hardware del IMU externo;
  \item necesidad de ventanas largas para obtener métricas defendibles.
\end{itemize}

\section{Casos de uso principales}

Los casos de uso que estructuran el sistema son:

\begin{itemize}
  \item captura y procesado en modo móvil;
  \item conexión y consulta de estado del modo ESP32;
  \item descarga y reprocesado de sesiones externas;
  \item revisión de resultados y comparación entre plataformas.
\end{itemize}

\section{Enfoque de trabajo}

Se adopta una \textbf{metodología de desarrollo incremental e iterativa}.
La razón de esta elección es que el trabajo combina varias fuentes de incertidumbre: disponibilidad real de sensores, diferencias entre plataformas, integración hardware progresiva y necesidad de validar el procesado antes de cerrar la instrumentación final.

En lugar de plantear un desarrollo lineal cerrado, el proyecto se organizó como una sucesión de iteraciones con entregables funcionales.
Cada iteración debía dejar operativo, al menos de forma parcial, uno de los bloques relevantes del sistema y producir evidencia suficiente para decidir el siguiente paso.

De forma resumida, la estrategia de trabajo por bloques funcionales fue la siguiente:

\begin{enumerate}
  \item Captura IMU y normalización temporal.
  \item Procesado espectral on-device.
  \item Visualización y exportación de sesión.
  \item Modo display para integración con ESP32.
  \item Validación comparativa final entre plataformas.
\end{enumerate}

Este enfoque incremental tiene varias ventajas en el contexto del TFG:

\begin{itemize}
  \item permite validar hipótesis del pipeline antes de depender del hardware definitivo;
  \item facilita aislar errores de adquisición, de procesado o de integración;
  \item evita que los problemas de una rama del sistema bloqueen el avance del conjunto;
  \item produce versiones parciales ya defendibles desde el punto de vista ingenieril.
\end{itemize}

\section{Fases del proyecto}

\begin{itemize}
  \item \textbf{Fase 1.} Definición de arquitectura del producto y formato común de datos.
  \item \textbf{Fase 2.} Prototipado de captura/procesado con scripts de referencia para reducir riesgo técnico.
  \item \textbf{Fase 3.} Integración on-device en Android y diseño de interfaz operativa.
  \item \textbf{Fase 4.} Integración ESP32+MPU6050 y modo Android display.
  \item \textbf{Fase 5.} Campaña comparativa y elaboración de conclusiones.
\end{itemize}

\section{Plan de pruebas y control de riesgos}

Los principales riesgos y su mitigación son:

\begin{itemize}
  \item Ventanas insuficientes para estabilidad espectral: imponer ventana robusta mínima.
  \item Variabilidad de frecuencia de muestreo: medir \(f_s\) real por sesión y registrar jitter.
  \item Diferencias de formato Android/ESP32: fijar esquema de salida común desde el inicio.
  \item Deriva o ruido excesivo: incluir validación de calidad y descarte de sesiones no válidas.
\end{itemize}

\section{Presupuesto orientativo}

El trabajo prioriza bajo coste de hardware:

\begin{itemize}
  \item teléfono Android: plataforma principal;
  \item ESP32 con LoRa: disponible en laboratorio personal;
  \item IMU MPU6050: disponible para fase comparativa.
\end{itemize}

El coste principal del proyecto se concentra en tiempo de desarrollo, integración y validación experimental.
