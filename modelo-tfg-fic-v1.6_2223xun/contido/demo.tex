\chapter{Desarrollo del sistema y validación}
\label{chap:demo}

\lettrine{E}{ste} capítulo describe el flujo técnico definido para el TFG y su implementación actual, alineada con el objetivo de usar Android como plataforma principal de captura y procesado inercial.

\section{Arquitectura funcional}

La arquitectura se diseñó con dos modos de operación en una misma aplicación móvil:

\begin{itemize}
  \item \textbf{Modo Android autónomo}: el teléfono captura sensores y procesa localmente.
  \item \textbf{Modo Android display}: el teléfono recibe resultados ya procesados desde ESP32+MPU6050 y actúa como interfaz de visualización y exportación.
\end{itemize}

\section{Flujo funcional definido}

El flujo del sistema se organiza en bloques:

\begin{enumerate}
  \item \textbf{Captura}: lectura de \texttt{accelerometer} y \texttt{gyroscope}, con soporte recomendado para \texttt{rotation vector}.
  \item \textbf{Normalización}: homogeneización de timestamps y estimación de frecuencia efectiva de muestreo.
  \item \textbf{Procesado}: método espectral tipo OpenMetBuoy.
  \item \textbf{Visualización}: gráfica de olas \(h(t)\) y panel de métricas.
  \item \textbf{Exportación}: almacenamiento de sesión con datos, métricas y parámetros.
\end{enumerate}

\subsection{Apartado IMU dentro del producto}

El módulo IMU es un componente central de la plataforma y se trata como parte explícita del producto:

\begin{itemize}
  \item \textbf{Entrada}: adquisición de acelerómetro y giroscopio con marcas temporales consistentes.
  \item \textbf{Normalización}: control de frecuencia efectiva y detección de irregularidades de muestreo.
  \item \textbf{Salida}: datos estructurados para procesado y exportación en formato común.
\end{itemize}

Esta separación en módulo permite mantener trazabilidad técnica del hardware inercial sin perder el foco principal del TFG, que es la plataforma móvil de captura y procesado.

\section{Pipeline de procesado adoptado}

El método seleccionado es un pipeline espectral tipo OpenMetBuoy, aplicado a la aceleración vertical:

\begin{enumerate}
  \item Estimación de \(\,f_s\,\) a partir del vector temporal.
  \item Separación de gravedad para obtener aceleración dinámica vertical.
  \item Estimación de PSD de aceleración mediante Welch.
  \item Conversión de espectro de aceleración a espectro de elevación:
  \[
    S_\eta(f) = \frac{S_a(f)}{(2\pi f)^4}
  \]
  \item Cálculo de momentos espectrales y métricas:
  \[
    H_s = 4\sqrt{m_0}, \qquad
    T_z = \frac{1}{\sqrt{m_2/m_0}}, \qquad
    T_c = \frac{1}{\sqrt{m_4/m_2}}
  \]
\end{enumerate}

Este enfoque está bien establecido en análisis de señales y caracterización espectral~\cite{Oppenheim2010}.

\section{Estrategia temporal de cálculo}

La validación realizada mostró una diferencia clara entre dos regímenes:

\begin{itemize}
  \item \textbf{Ventanas largas} (\(10\)--\(20\) min): resultados estables y repetibles para \(H_s\), \(T_z\), \(T_c\).
  \item \textbf{Ventanas cortas} (por ejemplo \(30\) s): útiles para diagnóstico rápido, pero no para estimación robusta principal.
\end{itemize}

Por tanto, la estimación principal del TFG se fundamenta en ventanas largas, mientras que el modo rápido queda como soporte operativo.

\section{Comparativa Android vs ESP32+MPU6050}

La fase de comparación se plantea con mismo protocolo físico y mismo formato de salida en ambas plataformas para evitar sesgos.
El flujo de comparación se resume en la Figura~\ref{fig:plantilla-comparativa}.

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.62\textwidth]{imaxes/esquema_3_comparativa.png}
  \caption{Flujo de comparativa entre plataformas con métricas unificadas.}
  \label{fig:plantilla-comparativa}
\end{figure}

Las métricas a contrastar son:

\begin{itemize}
  \item \(H_s\), \(T_z\), \(T_c\) y estabilidad temporal.
  \item Frecuencia efectiva de muestreo y jitter.
  \item Latencia de cálculo/visualización.
  \item Ruido residual y robustez del resultado.
  \item Consumo energético por sesión.
\end{itemize}
